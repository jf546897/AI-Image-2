# 安全策略

本项目是一个可便携运行的本地图像生成 Web 应用，安全问题主要集中在**凭证管理、日志脱敏、第三方提供方兼容差异**。

## 1. 支持范围

当前默认仅保证以下范围会被持续关注：

- 默认分支上的最新代码
- 当前便携目录结构对应的最新发布内容

历史快照、个人二次分发包或自行改造版本，可能不会得到同等优先级支持。

## 2. 报告安全问题前请先确认

以下情况通常属于安全问题：

- API Key、Token、Cookie 或本地配置可能被意外泄露
- 本地文件读写边界异常
- 未授权访问、越权操作
- 启动器、日志或配置流程会暴露敏感信息
- 第三方提供方兼容处理导致敏感数据被错误发送

以下情况通常**不属于**安全问题，而更适合走普通 Issue：

- 某个提供方不兼容但没有敏感数据风险
- UI 文案不准确
- 普通功能缺失

## 3. 报告方式

请**不要**在公开 Issue 中直接披露以下内容：

- 真实 API Key
- 可直接利用的完整攻击步骤
- 包含敏感头、用户数据或内部地址的原始日志

推荐流程：

1. 如果仓库已启用 GitHub 私密漏洞通道，请优先使用；
2. 如果没有私密通道，请先通过最小公开信息联系维护者，说明“需要私下报告安全问题”；
3. 在获得私密沟通方式前，不要公开贴出敏感细节。

## 4. 报告时建议包含的信息

请尽量提供：

- 问题类型与影响范围
- 受影响版本 / 提交号
- 最小复现步骤
- 预期行为与实际行为
- 已做过的脱敏日志或截图
- 你对风险等级的初步判断

## 5. 凭证与日志建议

为减少安全风险，请遵守以下原则：

- 不要提交 `local_config.json`
- 不要把真实密钥写进截图、Issue、PR 或日志
- 共享日志前先删除 Authorization、Cookie、Token、内部主机名等字段
- 对第三方网关做兼容测试时，优先使用专用测试密钥

## 6. 修复响应原则

维护侧通常会按以下顺序处理：

1. 先确认是否可复现；
2. 再确认是否存在真实安全影响；
3. 优先修复会导致敏感信息泄露或未授权访问的问题；
4. 在可行范围内补充文档、验证步骤或回归说明。

如果某问题暂时无法立即修复，至少会尽量给出缓解建议或风险说明。
